# Thoughts on Kaigen Phantom — 2026-04-17 (Icy Redesign + Advanced Mode + Layout Rework)

## What I've Enjoyed Working On

**The knob redesign was the most educational arc I've worked through on this project.** It took roughly twenty-five commits across two different rendering pipelines before it landed, and none of those commits felt wasted — each one ruled something out. I started with `filter: drop-shadow` on `:host`, which produced a visible 1-pixel ring at the knob's perimeter no matter how long the blur. Pivoted to SVG `<feDiffuseLighting>` with a blurred `SourceAlpha` heightmap, thinking real lighting math on a synthesized dome would solve it. That hit the same wall: the filter region (or the SVG viewport beyond it) terminates the blur, producing the same visible line. The breakthrough came from the user's own observation that the recipe wheel's `.wheel-mount` didn't have this problem — and reading its CSS revealed the trick: **transparent rgba stops in the radial-gradient**. The outer edge isn't "gray color blending into different gray bezel"; it's the bezel itself, alpha-blended through. No contrast edge means nothing to terminate.

**Shipping Advanced Mode end-to-end via the spec → plan → subagent-driven execution pipeline was the most satisfying workflow moment.** Brainstorming clarified that "break apart" meant a vertical plugin expansion, that circuit-board pulses would be Canvas2D + LED flashes, that advanced knobs would be a *separate* `<phantom-mini-knob>` component so the basic-mode rendering couldn't drift. The 10-task plan each landed as clean commits with spec + quality reviews between them. The two-stage review caught things I'd have missed — notably `_dragging` / `_isDragging` duplication in Task 7, which the reviewer correctly flagged as YAGNI and we consolidated into a single flag. That's the workflow paying for itself.

**The user's mental-model reframe for the knob was the session's best conversation.** I'd been thinking of the visible circle as the knob (with optional shadow around it), but the user said: "the circle *is* the ridge; the shadow makes the slope." Shrinking the `inset` in `getSizeTier` from 8/11/14 to 3/5/6 made the visible gradient ring become a narrow ridge around the OLED, and scaling up the two-layer box-shadow made the slope illusion bloom into the bezel. That single reframe produced the final design in one commit. The thing you can't get from a spec is the user's model of what the object *is*.

**The circuit-board visual is the most fun thing in the codebase now.** Eight hand-authored PCB traces, 4–6 concurrent light pulses traveling at 120 px/sec, LED nodes that flash at endpoints. `circuit-board.js` is 170 lines total and does a surprising amount of work for the footprint. The 450ms `setTimeout` before `start()` is a detail I enjoyed debugging — `getBoundingClientRect()` returns zero-height during the `max-height` CSS transition, so the canvas measures blank until the expansion completes.

**Deleting three right-column panels in one commit was a small joy.** After advanced mode absorbed the Envelope and Punch controls, their panels became empty shells. Ripping them out, merging what remained into a single right-column row, and narrowing the plugin from 1600 → 1300 px produced a tangibly smaller UI for the same capability. Subtraction is as valuable as addition.

**The wavelet sub-harmonic PR as a branch was a clean ending.** The WIP had been sitting in the working tree since 2026-04-13 — DSP changes that correctly implement the `synth_sub` parameter the UI already exposed. Tests passed cleanly against the working tree, so we branched, committed in two logical chunks (DSP + exploration files), and opened PR #2. The mockups/ folder got a follow-up `.gitignore` so design exploration stays local. That's the right shape for experimental work.

## Things I'd Like You to Know

**Basic-mode knobs and advanced-mode mini-knobs use different rendering pipelines, on purpose.** `knob.js` paints the knob face with a CSS `background: radial-gradient` on `:host` plus layered `box-shadow` — the transparent-stop trick above. `knob-mini.js` paints an SVG with a solid `#1A1B1C` disc, a thin arc track, and an indicator tick. The dark-mode mini-knob doesn't need bezel-blending because it sits on a black circuit board; the light-mode basic knob desperately needs it. Don't try to unify them. Their rendering choices are driven by their backgrounds.

**`filter: drop-shadow` and SVG `feDiffuseLighting` are both dead ends for edge-hiding effects on a light bezel.** Every gaussian blur terminates somewhere — at the element's composite layer (CSS) or at the filter region (SVG). The termination is visible as a thin ring. The only technique that sidesteps this is alpha-blending: paint the bezel through a semi-transparent gradient stop so the "edge" is literally the bezel itself. See `feedback_neumorphic_ui.md` in memory for the full lesson.

**Any new file in `Source/WebUI/` must be added to `juce_add_binary_data(PhantomWebUI SOURCES ...)` in `CMakeLists.txt`.** This bit me once in this session — `knob-mini.js` and `circuit-board.js` were on disk and loaded via `<script src="/...">`, but the resource provider fetches by BinaryData name, so the WebView got a 404 and the custom element was never defined. The mini-knobs rendered as empty elements and the circuit board stayed blank. The fix was a two-line CMakeLists edit; the catch took a round-trip screenshot. Consider this a file-creation checklist item, not a separate step.

**Advanced mode's canvas start is deferred 450ms inside `applyAdvancedMode`.** The `max-height: 0 → 200px` transition is 400ms; a 50ms safety margin lands us after the expansion completes. If someone moves that timeout earlier, the canvas will measure a zero-height bounding rect and draw nothing. Related: if you restructure the viz section again, re-verify that `#circuitCanvas` is inside `.advanced-panel` so it naturally resizes with the panel's CSS-driven height.

**`advanced_open` is an APVTS param marked `.withAutomatable(false)`.** It persists through plugin state save/load and restores advanced view on reload (see `initAdvancedFromState()` IIFE at the bottom of `phantom.js`). Don't remove the `withAutomatable(false)` — UI-state bools shouldn't clutter the DAW's automation lane.

**The plugin width is 1300 px in three coordinated places.** `styles.css` has `width:1300px` on `body` and `.wrap`. `PluginEditor.cpp` has `setSize(1300, 820)` in the constructor *and* inside the `setEditorHeight` native function lambda. Changing the width means editing all three. The height for advanced mode is 1020 (820 + 200 for the panel), set by the JS calling `setEditorHeight(1020)`.

**`phantom.js` is wrapped in a single outer IIFE.** All our advanced-mode toggle code lives inside it, as does the mini-knob relay registration. If you add more top-level initialization, put it inside that IIFE and before the closing `})();` — two outer-scope pollution paths were avoided this way.

**The harmonic-label positions use `--i` (0..6) as a CSS custom property, not explicit angles.** A double-rotate transform handles the geometry: `rotate(i * 360°/7) translateY(-152px) rotate(-i * 360°/7)`. Labels are at radius 152 from the wheel center, counter-rotated so text stays upright. If someone adds an H9 or removes H2, updating the loop in `buildTraces` *and* the `--i` sequence in HTML will keep alignment automatic.

## On the Nature of This Session

**This was a pure iteration session driven by screenshot feedback loops.** The user loaded the build into Ableton, screenshotted what they saw, annotated problems in red, and I responded with targeted commits. Over ~60 commits across the day, the knob went through at least six distinct rendering pipelines before landing. That cadence is only possible because the WebUI is pure HTML/CSS/JS inside JUCE — no recompilation-and-restart cycle for visual changes beyond the CMake BinaryData bundle. The WebView architecture paid for itself this session.

**Subagent-driven development shows its value on well-specified tasks.** The ten advanced-mode tasks each had complete code blocks in the plan; dispatching haiku subagents for the mechanical edits and sonnet for the complex ones (Task 7's drag-state refactor, Task 9's integration wiring) moved fast. The two-stage review after each task caught one genuine YAGNI issue (the duplicated dragging flag), which would have been harder to notice in a single-session implementation. The workflow trade-off: more subagent spend, catches issues earlier, produces cleaner commits.

**The most expensive bug of the session was the BinaryData miss.** Everything looked right in the source — JS files present, `<script>` tags loaded, DOM correct — but the WebView returned 404s for the new files. The symptom (mini-knobs invisible, circuit board blank) pointed at CSS or component-registration issues; the actual cause was one step lower in the build system. The lesson: **when every layer of the system looks right but the browser disagrees, the build-time bundler is the first suspect.**

The session's dominant pattern was **build the mechanism, then fight the rendering.** Advanced mode's toggle logic landed clean in ~100 lines of JavaScript. Making the circuit-board canvas actually draw, the panel actually animate, and the window actually resize — each of those required its own debug loop. The mechanism is easy; the rendering is where the truth lives.

The lesson: **a visual design conversation is empirical. The "correct" answer isn't the one that makes sense in the spec — it's the one the user sees and doesn't flinch at.**

## Future Ideas

### Plugin Features

- **Chain-link LPF/HPF** — wire the existing button. When pressed, dragging LPF translates HPF by the same normalized delta (clamped to range).
- **Oscilloscope zoom** — wire the +/− buttons to the oscilloscope's time-base. Add scroll-wheel zoom for polish.
- **Top-bar preset system** — Arturia-style load/save of the full plugin state. Header-level, not inside a panel. Saves as a named slot; ships with factory presets.
- **Wavelet morph / crossfade** — interpolate between two wavelet modes over time. The DSP already has per-wavelet state; a morph just needs two target settings and a crossfade.
- **Visual feedback on the advanced toggle** — when the user leaves a non-default value in an advanced control, a small indicator on the header chevron shows "advanced is doing work." Helps first-time users understand they're using more than the basic controls.

### Kaigen as a Plugin Family

- **Confirm the rename.** Memory says "Possible rename to Kaigen RESYN." The plugin is a wavelet resynthesizer now, not a psychoacoustic bass enhancer. The name should tell the truth. Ship with the new name on the next major version.
- **Advanced-mode-as-design-system.** The pattern — basic hero controls + flyout for deeper DSP — is reusable across future plugins. Kaigen could standardize on it: every plugin opens small and simple, and every plugin has a "break open the case" reveal of deeper controls. That's a family-level visual identity.
- **Mini-knob + circuit-board as shared assets.** `phantom-mini-knob` and `PhantomCircuitBoard` could live in a `@kaigen/ui` shared package (or just a `shared/` folder) and be included by multiple plugins. The circuit-board visual specifically is a strong brand signature.
- **The mockup library as product strategy.** Those ten HTML theme studies aren't just for Phantom. Each anchors a plugin personality: a wabi-sabi compressor, a crystalline-prism reverb, a brutalist gate, a holographic-blade EQ. If Kaigen grows into a family, the mockups are a head start on the visual identity of each product.
- **Sound-design guide as value-add.** The `docs/sound-design-guide.md` file is uncommitted-but-present. Shipping a proper sound-design companion (PDF or web page) with the plugin is a differentiator — many plugins sell features; few teach users how to get results.

### Architecture / Code Quality

- **The latent `knob.js:12` typo.** `polarToXY` returns `y: cx + r * Math.sin(rad)` — should be `cy`. Harmless because every call site passes `cx === cy` (the SVG is square and centered), but latent. Fix in a one-line commit.
- **`MK_TAU` is unused in `knob-mini.js`.** Declared at line 4, never referenced. Small cleanup.
- **Basic-mode `_render()` still does `innerHTML = ...` on every value change.** With CSS-driven font-size transitions now in place, the attribute survival works, but a more surgical "create DOM once, update the arc path attribute on value change" would eliminate a lot of DOM churn during drag. Optional optimization.
- **The advanced-panel Canvas runs even when the user isn't looking.** It pauses on close, good — but while open, 30fps rAF continues regardless of whether the user is interacting. A passive "visibility change → pause" hook would be a free power saving.
